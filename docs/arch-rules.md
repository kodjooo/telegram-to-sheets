# Правила разработки

Основные правила работы с проектом. Читать перед любой доработкой.

## Деплой и git

- Рабочая ветка — `main`. По завершении работ пушить в `https://github.com/kodjooo/telegram-to-sheets.git`.
- Использовать `origin` по **HTTPS** и push через HTTPS-аутентификацию, **не** через SSH-ключ.
- Папку `.claude/` в GitHub **не отправлять** (она в `.gitignore`).
- Секреты (`config.json`, `google-credentials.json`, `*.session`) в git не хранятся. Их нет в репозитории — на сервере они лежат в `./app` отдельно, `git pull` их не трогает.
- Информацию по развёртыванию на удалённом сервере писать в `README.md`.
- При изменении проекта обязательно актуализировать `docs/architecture.md`.

> Примечание: упоминание `.env.example`/`.env` и тестов `support-bot/tests`, `vector-base/tests` в `.claude/CLAUDE.md` относится к общему шаблону инструкций и в этом репозитории неприменимо — здесь нет ни `.env`, ни этих каталогов. Конфигурация ведётся через `app/config.json`.

## Развёртывание (важные нюансы)

- Контейнер поднимается `docker compose up -d --build`. Имя — `telegram-to-sheets-app`.
- **Папка `./app` примонтирована как volume** → изменения в `.py` подхватываются без пересборки.
- **`crontab` вшит в образ** (`COPY crontab` в `Dockerfile`). Любое изменение расписания требует **пересборки** (`--build`), простого `restart` недостаточно.
- Стандартный цикл деплоя на сервере (`/usr/local/other-scripts/telegram-to-sheets/`):
  ```bash
  git pull
  # при необходимости обновить app/config.json (он не в git)
  docker compose up -d --build
  ```
- Ручной запуск любого скрипта: `docker exec telegram-to-sheets-app python <script>.py`.

## Работа с сервером

- Для всего, что нужно посмотреть/проверить «на сервере» (логи, статус docker compose, файлы, конфиги, запущенные сервисы, рабочая БД) — подключаться по SSH под `developer` на `bkp.sellerdata.ru`, каталог `/usr/local/other-scripts/telegram-to-sheets/`, ключ `id_rsa`.

## Telethon-сессия — частые грабли

- Все telethon-скрипты используют **один** `session.session`. Чтобы не драться за SQLite-файл сессии:
  - Скрипты копируют сессию в свой `/tmp`-файл и работают с копией.
  - `telegram_to_sheets.py` копирует изменённую сессию **обратно** (персистит).
  - `alert_watcher.py` работает в режиме **только чтение** — обратно НЕ копирует, чтобы не конфликтовать с основным джобом.
- Есть retry на `database is locked` и на падения подключения (см. `TELEGRAM_*` константы).
- При добавлении нового telethon-скрипта повторять этот паттерн (своя tmp-копия, не писать обратно без необходимости).
- Аккаунт сессии должен состоять в группах, куда шлёт (`report_channel_id`, `alert_chat_id`), иначе отправка падает.

## Конвенции кода

- Пути захардкожены от `BASE_DIR = '/app'` (путь внутри контейнера). Учитывать при локальном запуске.
- Внешние вызовы (Google Sheets, Google API, Telegram) оборачивать в retry с backoff — образец `retry_gspread` / `retry_google_api` в `telegram_to_sheets.py`. Транзиентные статусы: `TRANSIENT_HTTP_STATUSES`, отдельная пауза на 429.
- Логи — через `logging` в свой файл в `/app/logs/`, формат `%(asctime)s - %(levelname)s - %(message)s`.
- Файлы состояния (`*_last_id.txt`, `*_state.json`, `*.lock`) лежат в `/app` и должны быть в `.gitignore`.
- Язык логов, сообщений и комментариев — русский, придерживаться его.

## Тестирование и документация

- Автотестов нет; проверка доработки — ручной запуск скрипта и просмотр соответствующего лога.

- Тестирование фронтенда (если появится) — через MCP Playwright.
